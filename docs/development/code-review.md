# コードレビューガイドライン

## 入力

レビュアー Agentは少なくとも次を確認する。

- `TASK.md`と受け入れ条件
- 承認済み`PLAN.md`
- `QA_PLAN.md`
- 対象ブランチの全差分とコミット
- DEVが固定した`candidate_commit`と`candidate_tree`、candidate-boundなHANDOVER証跡
- 変更範囲に適用される`AGENTS.md`
- 言語別コーディングガイドライン

## 観点

1. 正しさ: 正常系、境界値、失敗時、再試行、冪等性。
2. 責務境界: Plane、モジュール、永続化、外向き通信、権限。
3. 互換性: Schema、データ、API、CLI、設定、移行。
4. セキュリティ: 入力検証、秘密、認可、パス、コマンド実行。
5. 保守性: 命名、重複、複雑性、不要な抽象化、観測可能性。
6. テスト品質: 失敗を検出できるか、偽PASS、負例、境界統合。
7. Task整合: 受け入れ条件と完成の定義を根拠付きで満たすか。

## 重大度

- P0: 安全性、データ、権限、主要契約を壊し、直ちにマージを禁止する。
- P1: 正しさ、受け入れ条件、重大な回帰に関わり、マージ前修正を要する。
- P2: 非ブロッキングだが、追跡または改善理由を記録する。
- P3: 任意の改善提案。

## PASS条件

- P0とP1が0件である。
- P2を先送りする場合、非ブロッキングの根拠と追跡方法がある。
- `make check`が成功している。
- 変更に応じた専門検査が成功している。
- 各受け入れ条件に実装または試験の根拠がある。
- `REVIEW_RESULT.md`にレビュー対象コミット、コマンド、結果、残存リスクがある。
- QAと同一案から独立に開始し、QAのPASSを前提にしていない。
- 案の変更後は対象コミット/treeを更新し、旧案の結果を暗黙に再利用していない。

## 並行評価と修正

レビュアー AgentはQA Agentと同じ案を評価するが、相互のPASSを開始条件にしない。レビュー結果には対象`candidate_commit`/`candidate_tree`、実行コマンド、`make check`、差分のダイジェストと残存リスクを記録する。修正で案が変わった場合はMainが新案を確定し、QAへ再束縛する。Mainの`qa_carry_forward`記録がない限り、旧案のQA結果を新案へ移送しない。

指摘修正後は変更箇所だけでなく、同じ失敗モードで再レビューする。
